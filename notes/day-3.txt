CSS -> Cascading Stylesheet

1. inline style: directly applying style on a particulat element. not re-usable. best to avoid.
<element style="prop-name:value;prop-ame:value;">

2. outline style
create a separate CSS file and create styles there

a. element selector: name of all the elements on which you want to apply the set of styles. if the page contains multiple instances of the same element all of them will be styled same way.

el1-name, el2-name,...{
	/*styles*/
}

b. id selector: id of all the elements on which you want to apply the set of styles. if the page contains multiple instances of the same element with different id ONLY those elements matching to the mentioned ids will be styled.

el1-id, el2-id,...{
	/*styles*/
}

c. class selector: no need to mention any element name or id. just create a set of styles and then apply it on an element using "class" attribute
.class-name{
	/* styles */
}
<el class="class-name">

precedence: inline style > outline style > browser style



directives: extend the behavior of any existing element (html or custom)
------------------------------------------------------------------------
ngForm, ngModel (FormsModule), formGroup, formControl, formControlName (ReactiveFormsModule) -> attribute directives -> add additional behavior to the element without any modification to it

*ngFor/@for, *ngIf/@if, *ngIfElse/@if...else, *ngSwitch/@switch -> structural directives -> used to manipulate the DOM for the template

both structural and attribute directives are used as CUSTOM ATTRIBUTES of an element (that only Angular understands)

components -> component directive -> the ONLY directive with template/view. exposed NOT as an attribute, BUT as a CUSTOM HTML TAG (that only Angular understands)
@Component() will automatically apply @Directive on the component


class NgModel{
	control = new FormControl()
}


@Directive({
selector:'[ngForm]'})
export class NgForm{
	form = new FormGroup()
	form.controls={
		name1: new FormControl(),
		.....
	}
}

Service:
------------------------
where you can write any reusable piece of code for any business logic or data access logic or HTTP requests
you can persist state of data here which any component or other assets can use
sharing data amongst many assets of an angular application


1. create service
2. register a provider for the service

A. create a provider
	a. token name same as that of the service [default]
	b. a token name different from the name of the service [custom token based provider]

default provider object [default token] -> 
{
	provide:PostService,
	useClass: PostService
}

custom provider object [custom token] -> 
{
	provide:'POST_SERVICE',
	useClass: PostService
}


B. register the provider
	how/who you want to consume the service?
	a. a single component -> use 'providers' property of the @Component
	b. multiple components of the same feature module -> use 'providers' property of the @NgModule [feature module]
	c. root level ->
		i. modular app: use 'providers' property of the @NgModule [Root module/App module]
		ii. standalone app: use 'providers' property of the ApplicationConfig object in the 'app.config.ts'


HTTP requests:
------------------------
package: @angular/common/http
HttpClient -> built-in service

old: 
module: HttpClientModule (the provider for HttpClient class is registered here) -> deprecated

@NgModule({
providers:[{provide:HttpClient, useClass:HttpClient}]
})
export class HttpClientModule{}

new: instead of importing the HttpClientModule, call the function "provideHttpClient()" which will return the provider object for the service
providers:[
	provideHttpClient()
]

HttpClient asynchronous methods:
---------------------------------------------
	a. to send an HTTP request via GET method: get()
	b. to send an HTTP request via POST method: post()
	c. to send an HTTP request via PUT method: put()
	d. to send an HTTP request via DELETE method: delete()
	e. to send an HTTP request via PATCH method: patch()

all the methods return an Observable object
